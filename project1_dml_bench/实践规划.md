# DML-Bench：分布式优化算法模拟平台实践规划

## 一、当前阶段总目标

你现在第一步做 **DML-Bench Core 版**，不要一开始做成庞大的 Benchmark。目标是用 **7–10 天**先实现一个能跑通、能画图、能解释机制的分布式优化模拟平台。

本项目的核心不是追求复杂模型，而是把《分布式机器学习算法、理论和实践》中的关键机制写清楚、跑清楚、解释清楚。

| 项目 | 内容 |
|---|---|
| 项目名称 | DML-Bench：分布式优化算法模拟平台 |
| 第一版定位 | 单机模拟多 worker，不做真实多机通信 |
| 数据集 | MNIST |
| 模型 | Logistic Regression / 简单 MLP |
| 核心算法 | Centralized SGD、Sync-SGD、Local SGD、Async-SGD |
| 核心实验 | 通信频率、straggler、staleness、通信量统计 |
| 暂不实现 | DDP、Parameter Server、真实多进程、CIFAR-10、复杂 CNN |
| 最终输出 | 代码仓库 + CSV 日志 + 曲线图 + 小报告 |

### 第 3 阶段：实现 Sync-SGD

| 步骤 | 目标 | 需要实现的文件 | 具体任务 | 验收标准 |
|---|---|---|---|---|
| 3.1 | 广播全局模型 | `algorithms/sync_sgd.py` | 每轮把 server 模型复制给所有 worker | 每个 worker 初始参数一致 |
| 3.2 | worker 计算梯度 | `core/worker.py` | 每个 worker 在本地 batch 上计算梯度 | 能返回梯度列表 |
| 3.3 | server 聚合梯度 | `core/server.py` | 对所有 worker 梯度做平均 | 聚合梯度维度与模型参数一致 |
| 3.4 | server 更新模型 | `algorithms/sync_sgd.py` | 用平均梯度更新全局模型 | loss 能下降 |
| 3.5 | 统计通信量 | `utils/comm.py` | 记录每轮广播模型和上传梯度通信量 | CSV 中有 `comm_bytes` |
| 3.6 | 对比单机基线 | `scripts/run_sync.sh` | 跑 Centralized vs Sync-SGD | 两条 accuracy 曲线趋势接近 |

核心公式：

$$
g_t=\frac{1}{K}\sum_{k=1}^{K}g_k(w_t)
$$

$$
w_{t+1}=w_t-\eta g_t
$$

你需要理解：Sync-SGD 本质上是多个 worker 并行计算梯度，然后同步平均。

### 第 4 阶段：实现 Local SGD / Model Averaging

| 步骤 | 目标 | 需要实现的文件 | 具体任务 | 验收标准 |
|---|---|---|---|---|
| 4.1 | 本地训练多步 | `algorithms/local_sgd.py` | 每个 worker 本地训练 $E$ step | 本地模型参数发生变化 |
| 4.2 | 上传本地模型 | `core/worker.py` | worker 返回训练后的模型参数 | server 能接收多个模型 |
| 4.3 | 模型平均 | `core/server.py` | 对多个 worker 模型参数做平均 | 得到新的全局模型 |
| 4.4 | 设置本地步数 | `configs/local_sgd.yaml` | 支持 $E=1,2,5,10,20$ | 配置可切换 |
| 4.5 | 通信频率实验 | `scripts/run_local_steps.sh` | 比较不同 $E$ 下结果 | 生成 `local_steps_vs_acc.png` |
| 4.6 | 记录通信轮数 | `logger.py` | 保存 `communication_round` | 能画通信轮数-精度曲线 |

核心公式：

$$
w_k^{r,e+1}=w_k^{r,e}-\eta g_k(w_k^{r,e})
$$

$$
w^{r+1}=\frac{1}{K}\sum_{k=1}^{K}w_k^{r,E}
$$

你要重点观察：本地训练步数 $E$ 增大后，通信减少，但模型漂移可能增强。

### 第 5 阶段：实现 Async-SGD

| 步骤 | 目标 | 需要实现的文件 | 具体任务 | 验收标准 |
|---|---|---|---|---|
| 5.1 | 设计模型版本号 | `core/server.py` | 每次 server 更新后 `version += 1` | 能记录当前全局版本 |
| 5.2 | worker 拉取旧模型 | `core/worker.py` | 记录 worker 拉取模型时的版本号 | 返回 `pull_version` |
| 5.3 | 实现事件队列 | `core/simulator.py` | 用虚拟时间模拟 worker 完成顺序 | 不使用真实 `sleep` |
| 5.4 | 异步提交梯度 | `algorithms/async_sgd.py` | worker 完成后立即上传梯度 | server 不等待所有 worker |
| 5.5 | 计算 staleness | `algorithms/async_sgd.py` | `staleness = current_version - pull_version` | CSV 有 `staleness` |
| 5.6 | 异步更新模型 | `core/server.py` | server 收到一个梯度就更新一次 | loss 能下降 |
| 5.7 | 对比 Sync-SGD | `scripts/run_async.sh` | 同样设置下比较 Sync 和 Async | 能生成两条曲线 |

核心公式：

$$
w_{t+1}=w_t-\eta g_k(w_{t-\tau})
$$

其中 $\tau$ 是 staleness。

你要重点观察：Async-SGD 不等慢节点，但旧梯度可能影响稳定性。

## 九、当前版本的完成标准

你第一版 DML-Bench 完成时，至少要能回答这几个问题：

| 问题 | 你应该能给出的答案 |
|---|---|
| Sync-SGD 和 Centralized SGD 有什么关系？ | Sync-SGD 是多 worker 梯度平均，本质类似更大 batch 的 SGD |
| Local SGD 为什么减少通信？ | 因为本地训练 $E$ 步后才上传模型 |
| Local SGD 的风险是什么？ | $E$ 过大时 worker 模型漂移 |
| Async-SGD 的优势是什么？ | 不等待慢 worker |
| Async-SGD 的风险是什么？ | stale gradient 导致更新方向滞后 |
| straggler 为什么影响同步训练？ | 同步屏障要求所有 worker 到齐 |
| 为什么要统计通信量？ | 分布式训练的核心权衡是精度、时间和通信成本 |
