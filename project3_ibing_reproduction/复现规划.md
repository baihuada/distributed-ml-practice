## 结论

你接下来复现 IBing，不建议一上来直接复现论文全部大规模实验。正确路线应该是：

**先复现算法正确性 → 再复现通信调度 → 再复现 MPI 多进程通信 → 最后做性能对比与论文趋势验证。**

完整复现可以分为 **6 个阶段**：

1. **阶段 0：明确复现目标与边界**
2. **阶段 1：单进程模拟 IBing 通信调度**
3. **阶段 2：实现标准 Ring All-Reduce baseline**
4. **阶段 3：实现 MPI 版 IBing**
5. **阶段 4：做通信性能 benchmark**
6. **阶段 5：接入分布式训练或模拟训练场景**
7. **阶段 6：整理复现实验报告**

论文原始实验是在天河 MT-3000 超算集群、OpenMPI 4.0.3、最多 1024 节点上完成的，所以你本地或普通服务器复现时，重点应该放在 **算法正确性、通信步数、性能趋势**，不必强行追求完全相同的 49.73% 最高优化率。论文核心依据是 IBing 将 Ring All-Reduce 的通信步数从 \(2(N-1)\) 降到 \(N-1\)，并通过双向交错通信降低通信建立与数据处理开销。

### 1.3 先实现 rank 与邻居计算

对于第 \(r\) 个 worker：

$$

$$

def get_neighbors(rank: int, world_size: int):

    right = (rank + 1) % world_size

### 1.4 实现 IBing 的 chunk 调度公式

对于当前 rank \(r\)，step \(i\)，论文中的四个核心 chunk 公式是：

$$
\texttt{recv\_chunk\_1}=(r-i-1+N)\bmod N
$$

$$
\texttt{send\_chunk\_1}=(r-i+N)\bmod N
$$

$$
\texttt{recv\_chunk\_2}=(r+i+N+2)\bmod N
$$

$$
\texttt{send\_chunk\_2}=(r+i+N+1)\bmod N
$$

你应该写成函数：

```python
def ibing_schedule(rank: int, step: int, world_size: int):
    recv_chunk_1 = (rank - step - 1 + world_size) % world_size
    send_chunk_1 = (rank - step + world_size) % world_size

    recv_chunk_2 = (rank + step + world_size + 2) % world_size
    send_chunk_2 = (rank + step + world_size + 1) % world_size

    return {
        "recv_chunk_1": recv_chunk_1,
        "send_chunk_1": send_chunk_1,
        "recv_chunk_2": recv_chunk_2,
        "send_chunk_2": send_chunk_2,
    }
```

### 2.2 Ring All-Reduce 的基本步骤

标准 Ring All-Reduce 分两阶段。

### 第一阶段：Reduce-Scatter

需要：

$$
N-1
$$

步。

每一步每个 worker 向右发送一个 chunk，从左接收一个 chunk，并做 reduce 加法。

------

### 第二阶段：All-Gather

也需要：

$$
N-1
$$

步。

每一步每个 worker 继续向右发送一个已经聚合完成的 chunk，从左接收一个 chunk，但不再做加法。

------

### 2.3 Ring 的总步数

$$
\mathrm{steps}_{\mathrm{Ring}}=2(N-1)
$$

IBing 的总步数是：

$$
\mathrm{steps}_{\mathrm{IBing}}=N-1
$$

所以你后面报告中要突出这一点：

| 算法             | 通信方向 | 总步数   |
| ---------------- | -------- | -------- |
| Ring All-Reduce  | 单向     | \(2(N-1)\) |
| IBing All-Reduce | 双向交错 | \(N-1\)    |

## 3.3 MPI 版实现重点

每个 MPI 进程对应一个 worker。

你需要获取：

```python
from mpi4py import MPI

comm = MPI.COMM_WORLD
rank = comm.Get_rank()
world_size = comm.Get_size()
```

左右邻居：

```python
left = (rank - 1 + world_size) % world_size
right = (rank + 1) % world_size
```

每个进程初始化本地数组：

```python
data = np.full((world_size, chunk_size), fill_value=rank, dtype=np.float32)
```

这里 `data[chunk_id]` 表示当前 rank 保存的第 `chunk_id` 个 segment。

------

## 3.4 通信方式

mpi4py 里可以使用非阻塞通信：

```python
req1 = comm.Isend(send_buf_1, dest=right, tag=tag1)
req2 = comm.Irecv(recv_buf_1, source=left, tag=tag1)

req3 = comm.Isend(send_buf_2, dest=left, tag=tag2)
req4 = comm.Irecv(recv_buf_2, source=right, tag=tag2)

MPI.Request.Waitall([req1, req2, req3, req4])
```

注意两点：

1. 最好先 `Irecv`，再 `Isend`；
2. 左右两个方向的 tag 要区分，避免消息错配。

例如：

```python
tag_lr = step * 2
tag_rl = step * 2 + 1
```

------

## 3.5 MPI 版 IBing 的核心循环

伪代码如下：

```python
for step in range(world_size - 1):
    recv_chunk_1 = (rank - step - 1 + world_size) % world_size
    send_chunk_1 = (rank - step + world_size) % world_size

    recv_chunk_2 = (rank + step + world_size + 2) % world_size
    send_chunk_2 = (rank + step + world_size + 1) % world_size

    send_buf_1 = data[send_chunk_1].copy()
    send_buf_2 = data[send_chunk_2].copy()

    recv_buf_1 = np.empty_like(send_buf_1)
    recv_buf_2 = np.empty_like(send_buf_2)

    req_recv_1 = comm.Irecv(recv_buf_1, source=left, tag=step * 2)
    req_recv_2 = comm.Irecv(recv_buf_2, source=right, tag=step * 2 + 1)

    req_send_1 = comm.Isend(send_buf_1, dest=right, tag=step * 2)
    req_send_2 = comm.Isend(send_buf_2, dest=left, tag=step * 2 + 1)

    MPI.Request.Waitall([req_recv_1, req_recv_2, req_send_1, req_send_2])

    if step < reduce_steps:
        data[recv_chunk_1] += recv_buf_1
        data[recv_chunk_2] += recv_buf_2
    else:
        data[recv_chunk_1] = recv_buf_1
        data[recv_chunk_2] = recv_buf_2
```

这里 `reduce_steps` 需要你特别小心。论文中对奇偶节点的划分有说明：当 \(N=5\) 时，前 2 步是 reduce，后 2 步是 allgather；当 \(N\) 为偶数时，需要根据论文中 `round((N-1)/2)` 的逻辑处理。你第一版可以先从 **奇数节点 \(N=5\)** 做通，再扩展到 \(N\in\{4,8,16\}\)。

# 关键风险与应对方案

## 风险 1：IBing 奇偶节点阶段划分容易写错

论文中 \(N=5\) 的图最直观，建议你先只做：

$$
N=5
$$

确认完全正确后，再扩展：

$$
N\in\{4,8,16\}
$$

不要一开始就覆盖所有情况。

------

## 风险 2：单机多进程性能不代表真实多机通信

如果你在一台电脑上用 `mpirun -np 8`，通信大多发生在本机进程之间，和论文中的超算网络不同。

所以报告里要明确写：

```text
本复现实验主要验证 IBing 的算法正确性与通信调度趋势，实验环境不同于原论文的天河 MT-3000 大规模集群，因此不追求完全一致的绝对通信时间。
```

------

## 风险 3：MPI 非阻塞通信可能死锁或消息错配

应对方案：

1. 先 `Irecv`，再 `Isend`；
2. 左右方向使用不同 tag；
3. 每一步 tag 与 step 绑定；
4. 每个发送 buffer 使用 `.copy()`，避免还没发完就被覆盖；
5. 每一步 `Waitall` 后再进入下一步。

------

## 风险 4：IBing 和 Ring 结果不一致

排查顺序：

1. 检查 chunk 切分是否一致；
2. 检查 send/recv chunk 公式；
3. 检查左、右邻居是否写反；
4. 检查 Reduce-Scatter 阶段是否做了加法；
5. 检查 All-Gather 阶段是否误加法；
6. 检查 tag 是否匹配；
7. 检查发送的是 copy 还是原始引用。
